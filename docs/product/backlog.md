## 1. 每周模板"版本化 / 存档"（后端① + console②）

**需求**：每个孩子的每周模板可以有多个"方案 / 版本"，同一时刻一个"生效"，其余"存档"；孩子升学 / 进新阶段时新建方案（默认从当前复制当起点），旧的转存档、可重新启用。

**影响 / 难点**：改后端领域模型——新增 `WeeklyTemplate` 实体（`ChildId` + `Name`（如"三年级上"）+ `Status` 生效/存档 + 可选生效起始日），`WeeklyTaskTemplateItem` 从"挂孩子"改成"挂方案"；每日生成器（`get-board` 懒生成）改成读**当前生效方案**；EF 迁移。前端每周模板页顶部加"方案选择 + 新建 / 复制 / 重命名 / 存档 / 启用"。

**要点**：一个孩子同一时刻**恰好一个**生效方案；**历史数据不动**（已生成的每日任务 / 星星 / 记分是历史记录，换方案 / 存档只影响以后的生成）。

## 2. 无密码认证改造（后端① + console②）

**需求**：
- **注册 = 纯邮箱**：**邮箱即用户名**（去掉用户名字段）→ 系统发 **magic link** 到邮箱 → 点击激活。**注册阶段不输密码。**
- **登录两种方式**：① 密码登录；② 邮箱验证码（OTP）登录。
- **密码在登录进去之后再（可选）设置。**

**影响 / 难点**：ABP OpenIddict **默认没有** magic-link，也没有"邮箱验证码登录"——都得**自定义**：一次性 token / 验证码生成、发邮件、验证后换 JWT、以及"登录后补设密码"这条路（首次无 current password 的设密）。

**硬前提：系统得能真的发邮件。**
- **dev**：可先把链接 / 验证码**打印到控制台**，不真发信也能把整套流程跑通、联调。
- **上线前**：必须接真实邮件服务（自建 SMTP，或 Resend / SendGrid / 阿里云邮件推送 等）——**待选型**。

**设计待定（进 spec 时敲）**：magic link 与验证码的**有效期、单次使用**；验证码**限流防爆破**；**邮箱枚举**防护；ABP 侧用自定义 OpenIddict grant 还是自建端点发 token。

**注**：现有注册页（邮箱 + 密码 + 同意勾选）与 ROPC 密码登录会被这套替换 / 扩展；"孩子是家长名下档案"的归属模型不受影响。
